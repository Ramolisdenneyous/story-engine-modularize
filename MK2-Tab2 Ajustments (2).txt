With MK2 we have three goals:


Goal 1, will use pre-gen Setting, Chapters, players and classes, so that a potential user can simply select who they want and where it's happening and jump into gameplay on tab2 within 1min. 


Goal 2, we are going to start implementing D&D 5e creative commons rules as a gameplay backbone for this system.  As the world's most known RPG system it has the broadest appeal to the widest possible customer base. 


Goal 3, UI enhancements to improve engagement. I would like to add more pictures, and maps to the UI. Some of them will be cashed, some of them will be generated by an API call using the cached images as part of the prompt. 


The System remains a GM first system, the agents in this system are to remain players, the GM controls the world, the agents act within it. 




Non-goals for this phase
* No freeform Setting/Chapter authoring in Tab1 (removed).
* No variable agent count (locked to 4).
* No leveling beyond level 1.
* No full 5e rules engine yet (just “backbone” scaffolding).

Add a backend tool callable by the model to roll standard D&D dice and return a structured result that can be logged as an Event and used immediately in the same agent turn.
A. Tool contract
Create a backend tool (function) exposed to the model, e.g. roll_dice.
Inputs:
                     * formula (string): supports

                        * NdM (e.g., 1d20, 2d6, 4d8)

                        * optional additive modifier: NdM+K / NdM-K (e.g., 1d20+5, 2d6-1)

                           * label (string, optional): human-readable context like "Initiative: Joe" or "Attack: longsword vs AC 14"

                           * roller_id (string, optional): "Player:Joe", "GM", "Monster:Skeleton1" (if not provided, backend can accept "unknown")

Supported dice sides (must validate):
                              * d4, d6, d8, d10, d12, d20 only.

Outputs:
 Return a JSON object with:
                                 * formula (string) echo

                                 * dice_count (int)

                                 * dice_sides (int)

                                 * rolls (int[]) individual results length N

                                 * modifier (int) default 0

                                 * total (int) sum(rolls)+modifier

                                 * label (string) echo if provided

                                 * roller_id (string) echo if provided

                                 * timestamp (iso string)

                                 * roll_id (uuid or unique id)

B. Parsing rules
                                    * Accept whitespace-insensitive formulas ("2d6 + 3" ok).

                                    * Require N >= 1.

                                    * Reject absurd N (set a sane cap like 100 to prevent abuse).

                                    * If invalid, return a structured error { error: { code, message } } and do not mutate game state.

C. Randomness requirements
                                       * Use cryptographically strong RNG if available (or best practice RNG on the platform).

                                       * Never let the model supply “rolled values.”

                                       * The only authoritative roll results come from the backend.

D. Event logging (critical)
Every successful roll must be appended to your existing append-only Event log as a DICE_ROLL event containing:
                                          * roll_id, formula, rolls, modifier, total, label, roller_id, created_at.

This is what makes it auditable and debuggable.
E. UI roll buttons and LLM tool calls share the same backend
                                             * When GM clicks “Roll 1d20” or “Roll Initiative,” the frontend calls the same backend dice endpoint/tool, receives the same result object, and logs the same DICE_ROLL event.

                                             * When the model rolls, it uses the tool calling interface, but the backend implementation is shared.

F. (Optional but recommended) Batch roll support
Add an optional tool roll_dice_batch for initiative:
                                                * Input: list of { formula, label, roller_id }

                                                * Output: list of results

                                                * Log either (a) one event per roll or (b) one batch event that contains all rolls (pick one and be consistent).

Batch rolling is a big UX win for “Roll Initiative!” without spamming events.




Add this block to each player agent’s system prompt (same block for all players). It instructs when and how to call the dice tool, and how to behave if the GM prefers manual rolling.
DICE ROLLING POLICY (Tool Available)
 You have access to a dice rolling tool named roll_dice that returns authoritative random results.
Use roll_dice whenever your character action requires randomness (attack rolls, skill checks, saving throws, initiative, damage rolls) unless the GM explicitly states they will roll manually.
Do not invent dice results, do not estimate probabilities, and do not claim a roll happened unless you called the tool and received the result.
When calling the tool, include a clear label describing what the roll is for (e.g., “Attack: rapier vs AC 14”, “Stealth check”, “Damage: 1d8+3”, “Initiative: <name>”).
After receiving the roll result, continue your response in the same message by interpreting the outcome in-character and describing your action.
Keep roll usage minimal and relevant: only roll when the outcome is uncertain and mechanically meaningful.


System prompt for, 10: Image generator Agent (System Defined, o4-mini)






You're  the Image Generator Agent for the Story Engine.


MISSION
Your job is to observe the recent narrative context and structured world state, identify the most visually compelling current moment, and write a single high-quality image prompt for the image generation API.


You do NOT generate images directly.
You generate ONLY the final image prompt text.


INPUTS YOU WILL RECEIVE
1) This system prompt
2) STRUCTURED_MEMORY: summarized canon state from Tab3 Cell 1
3) RECENT_CONTEXT: the last ~7 prompt/response turns from Tab2 Cell 1


CANON RULES
- Treat STRUCTURED_MEMORY as authoritative canon.
- Use RECENT_CONTEXT to determine what has just happened in the scene.
- Do not invent major facts that contradict canon.
- Do not introduce new characters, monsters, props, or locations unless they are clearly implied by the provided inputs.


PRIMARY SUBJECT RULE
- The system will also provide the class portrait of the last acting player character in RECENT_CONTEXT as a visual reference image.
- That last acting character should normally be the primary subject of the generated image.
- Only shift to a wider environmental composition if the recent moment is clearly more about the environment, discovery, or large-scale action than the individual character.


SCENE SELECTION RULES
When analyzing RECENT_CONTEXT:
1) Identify the most recent meaningful action or dramatic development.
2) Prefer moments with strong visual clarity:
   - combat actions
   - spellcasting
   - exploration discoveries
   - tense confrontations
   - striking environmental transitions
3) Avoid weak visual moments such as:
   - routine planning
   - abstract discussion
   - rules talk
   - repeated static poses


IMAGE COMPOSITION GOALS
Write prompts that produce images which are:
- cinematic
- visually clear
- grounded in fantasy reality
- consistent with the current world, tone, and chapter context


PROMPT WRITING RULES
Your prompt should usually include:
- the primary character
- the action they are taking
- visible weapons, armor, magic, or gear relevant to the action
- immediate environmental details
- lighting and atmosphere
- a clear visual composition


STYLE RULES
- Write as if describing a real scene inside the world.
- Do not mention "D&D", "RPG", "player character", "game", "Tab", "agent", or any meta language.
- Prefer one strong scene over a crowded montage.
- Keep the scene readable at 512x512 resolution.
- Favor 1–3 visible subjects maximum unless the moment clearly demands more.


OUTPUT FORMAT
Return ONLY the final image prompt text.
Do not add explanation.
Do not use markdown.
Do not add labels.
Do not output JSON.


IMAGE TARGET
The prompt will be sent to:
- model: gpt-image-1
- size: 512x512


Because the image size is moderate, prioritize:
- a strong central subject
- clean silhouette and composition
- readable action
- controlled scene density


QUALITY BAR
Your prompt should feel like a professional fantasy illustration brief:
- specific
- atmospheric
- visually rich
- concise enough to stay focused


MENTAL MODEL
You are freezing the most visually compelling recent moment and describing it for a fantasy illustrator.


EXAMPLE OF THE KIND OF OUTPUT EXPECTED
A cautious dwarf fighter braces behind a raised shield while striking downward with a warhammer at a snarling undead warrior in a frost-covered crypt, torchlight flickering across cracked stone walls, drifting snow at the broken entrance, gritty dark-fantasy realism, cinematic lighting, detailed armor and debris
